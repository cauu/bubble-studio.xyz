過去幾年，當區塊鏈世界談論「組合性」時，預設指的往往都是 **Ethereum 式的同步呼叫**：合約像公開 API 一樣彼此直接呼叫，在同一筆交易中拼接出複雜行為。這種「樂高式」模式催生了 DeFi 的繁榮，也讓像 EIP-4337 這樣偏執行導向的 Agent 變得越來越強。

但如果我們把視角從「服務人類開發者」切換到「服務 AI Agent」，底層的優化目標就會發生根本性的改變。

如果未來的主要使用者是 AI，那麼系統最應該優化的，不是「執行自由度」，而是 **可預測性、可驗證性，以及明確的權限邊界**。順著這條邏輯往下推，我會得出一個可能讓很多人意外的結論：**AI-native 區塊鏈的底層帳本，在邏輯上應該更接近 Cardano，而不是 Ethereum。**

#### 1. 確定性比自由更重要

Ethereum 的強項，在於最大化執行時的自由度；但代價是執行過程中的不確定性。你發出一筆交易，直到真正被打包之前，你都未必能完全知道它會碰到哪些狀態、最終消耗多少 Gas。

對人類來說，這只是設計風格的差異；但對 AI 來說，這會帶來極大的效率損失。AI 最擅長的，其實是 **先做規劃、先驗證約束、再執行步驟**。

Cardano 的 eUTXO 模型天然契合這種思維方式。交易處理是確定性的。AI 可以在本地相當精準地預估一筆交易的影響，而不是把複雜性丟進執行中的黑盒裡，等結果出來才知道發生了什麼。對一個需要自我迭代、失敗重試、持續調整的工作流來說，這種「執行前就能算清楚」的架構，顯然比「執行時再看結果」友善得多。

#### 2. 「顯式編排」vs「隱式互調」

Ethereum 的合約彼此開放，A 呼叫 B，B 再呼叫 C，控制流程在執行時動態展開。這對高密度原子操作來說非常強大，但也讓風險邊界變得模糊。你在設計階段看到的可能只是一次介面呼叫，但在執行時，它可能引發一整串巢狀呼叫與狀態變更。

相比之下，Cardano 限制了合約之間的直接呼叫，迫使開發者把更多工作放到鏈下編排：選哪些輸入、滿足哪些驗證條件、如何建構交易。

很多人會把這看成表達能力的損失，但對 AI Agent 而言，這反而可能是優勢。因為 **複雜關係被顯式暴露出來了**。從 AI 的角度看，這更像是一個可以求解的工作流問題，而不是一個難以穿透的動態執行環境。AI 不怕規則嚴格，它怕的是邊界不清楚。

#### 3. 從「原子大招」走向可恢復工作流

Ethereum 追求的是 all-or-nothing 的原子性：把所有動作塞進一筆交易裡，要嘛一步完成，要嘛整筆回滾。

但 AI Agent 的典型工作邏輯，通常是長生命週期的：

- 讀取狀態 -> 生成計畫 -> 本地驗證 -> 執行步驟 -> 若失敗，再重新編排。

這種模式更接近分散式工作流，而不是單一的鏈上原子操作。Cardano 的 eUTXO 狀態本身就是離散資源；當交易失敗時，往往意味著前置條件變了，AI 只需要重新選擇輸入、重新規劃即可。這種 **可重試、可恢復** 的執行模式，比起把所有邏輯硬塞進一筆原子交易裡，更符合 AI 長期運作的直覺。

#### 4. 底層帳本本身就是「資源調度系統」

如果未來是成千上萬個 Agent 在協作，那麼系統真正的核心問題會變成：資源依賴如何宣告？哪些狀態可以並行？

Cardano 把狀態表達成離散資源（UTxO），而應用設計從一開始就被要求避免單一狀態點的競爭。這種底層邏輯，更像是一套 **資源調度與約束滿足系統**。

AI-native 區塊鏈不應該先問：「我能不能自由地 call 一切？」而應該問：「我能不能清楚地宣告資源，並明確處理衝突？」在這種哲學下，底層帳本更像是一個宣告式的約束系統，而不是單純的命令式執行環境。

#### 最後的收斂：一種融合的未來

當然，這並不代表 Cardano 已經贏了。

目前的 Cardano，在 Agent 權限模型、鏈下編排的標準介面，以及任務佇列的處理上，仍然有很長的路要走。而 Ethereum 生態在帳戶抽象（AA）與模組化執行上累積的經驗，依然是極其寶貴的資產。

我更傾向相信：**未來 AI-native 鏈的底層，會吸收 Cardano 式的確定性資源模型；而在帳戶層與中介層上，則會保留 Ethereum 長期積累下來的靈活性。**

所以，比起問「誰會取代誰」，更值得思考的是：當我們為機器社會建設基礎設施時，底層應該是一個隨意的執行環境，還是一個嚴謹的約束系統？

從 Aiken 這種把 property-based testing 視為一等公民的嘗試來看，答案正在變得越來越清晰。
